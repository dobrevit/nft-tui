# Data model

The TUI's in-memory model mirrors the nftables object hierarchy, plus a
staged-changes layer on top.

## nftables, in one diagram

```
Ruleset
└── Table         (family: ip | ip6 | inet | arp | bridge | netdev, name)
    ├── Chain    (type, hook, priority, policy, device — for netdev/ingress)
    │   └── Rule (handle, expressions, statements/verdicts, counter, comment)
    ├── Set      (type, flags: timeout, interval, dynamic, counter, constant)
    ├── Map      (key type → value type/verdict; same flag space as Set)
    ├── FlowTable (devices, hook, priority)
    └── Object   (counter | quota | limit | ct helper | ct expectation |
                   ct timeout | secmark | synproxy)
```

A *table* is a namespace. A *chain* belongs to one table and is either a
**base chain** (has `type`/`hook`/`priority`/`policy`, kernel attaches it
to a netfilter hook) or a **regular chain** (called by `jump`/`goto`).
Rules live in chains and have a numeric **handle** that's stable until
the rule is deleted.

## Go types (sketch — to be refined when coding starts)

```go
// internal/model/ruleset.go

type Family string
const (
    FamilyIP     Family = "ip"
    FamilyIP6    Family = "ip6"
    FamilyINet   Family = "inet"
    FamilyARP    Family = "arp"
    FamilyBridge Family = "bridge"
    FamilyNetdev Family = "netdev"
)

type Ruleset struct {
    Tables []*Table
    // metadata
    FetchedAt time.Time
    Source    string // "nft-json" | "netlink"
}

type Table struct {
    Family      Family
    Name        string
    Chains      []*Chain
    Sets        []*Set
    Maps        []*Map
    FlowTables  []*FlowTable
    Objects     []*Object
    Comment     string
}

type Chain struct {
    Table    *Table       // back-pointer (not serialised)
    Name     string
    // base-chain fields (all empty for regular chains):
    Type     string       // "filter" | "nat" | "route"
    Hook     string       // "input" | "output" | "forward" | ...
    Priority int
    Policy   string       // "accept" | "drop"
    Device   string       // for netdev / ingress
    Rules    []*Rule
    Comment  string
}

type Rule struct {
    Chain   *Chain        // back-pointer
    Handle  uint64        // 0 == not yet committed
    Exprs   []Expr        // structured AST (match clauses + statements)
    Counter Counter       // pkts/bytes, even if no explicit `counter` keyword
    Comment string
    // raw `nft` text representation; we always keep this as the source of truth
    // for display, even when we built the rule from a form.
    NFT string
}

type Counter struct {
    Packets uint64
    Bytes   uint64
    SeenAt  time.Time
}

type Set struct {
    Table    *Table
    Name     string
    KeyType  string        // "ipv4_addr", "inet_service", ...
    Flags    SetFlags
    Timeout  time.Duration // zero if not set
    Elements []SetElement
    Comment  string
}

type SetFlags struct {
    Constant bool
    Dynamic  bool
    Interval bool
    Timeout  bool
    Counter  bool
}

type SetElement struct {
    Key     string
    Expires time.Time     // zero if no timeout
    Counter Counter       // only meaningful if flag is set
    Comment string
}

// Map, FlowTable, Object — same idea.
```

The `Expr` interface is deliberately not designed in detail at this stage:
we will start by treating each rule's body as opaque `nft` text plus a
flat list of *recognised* expressions (`iifname`, `oifname`, `saddr`,
`daddr`, `sport`, `dport`, `ct state`, `tcp flags`, `verdict`) used to
power the form-based editor and the column view. Anything we don't
recognise still round-trips through `NFT`. **This is the safety contract:
unknown nftables features never get silently dropped on commit.**

## Source of truth

`nft -j list ruleset` is the input. Its JSON schema is documented in
`libnftables-json(5)`. We parse it into the model above. A rule's `NFT`
field is generated by re-rendering the JSON into the canonical text
representation (or, more reliably, by calling `nft list chain <fam> <tbl>
<chain>` on display — TBD during implementation).

## Staged changes

```go
// internal/state/staged.go

type StagedOp interface {
    // Render the op as `nft` syntax (one or more statements, newline-separated).
    NFT() string
    // Apply to a Ruleset to produce a projected post-commit view.
    Apply(*model.Ruleset) error
    Describe() string // for the tree view of staged changes
}

type AddRule    struct { Table, Chain string; Family model.Family; NFT string; After uint64 /* handle, 0 = append */ }
type DeleteRule struct { Table, Chain string; Family model.Family; Handle uint64 }
type ReplaceRule struct { Table, Chain string; Family model.Family; Handle uint64; NFT string }
type AddChain   struct { Table string; Family model.Family; Chain model.Chain }
type DeleteChain struct { Table, Chain string; Family model.Family }
type FlushChain struct { Table, Chain string; Family model.Family }
type AddSetElement    struct { Table, Set string; Family model.Family; Key string; Timeout time.Duration }
type DeleteSetElement struct { Table, Set string; Family model.Family; Key string }
// … etc.
```

A `StagedChange` slice is the only mutable state. Commit serialises every
op's `NFT()` into a temp file wrapped in:

```
add table inet filter   # idempotent guards if needed
...
```

…then runs `nft -c -f tmpfile` (dry-run) and finally `nft -f tmpfile`.
Both go through a single transaction in the kernel, so either everything
applies or nothing does.

## Refresh & change detection

After every commit, **and** on a ticker (default 2 s), the adapter re-runs
`nft -j list ruleset`. We diff the new `Ruleset` against the cached one
by `(family, table, chain, handle)` and update the UI without resetting
scroll positions.

If an external process changed the ruleset (someone ran `nft` in another
SSH session), the status bar shows a yellow "kernel changed — your
staged changes may conflict" warning, and the diff screen recomputes
against the new baseline.

## What we deliberately do NOT model

- Per-CPU counter breakdowns. The aggregate is enough for triage; if
  someone wants per-CPU they can drop to `nft list` or `cat
  /proc/net/nf_*`.
- Connection-tracking entries (`conntrack -L`). Separate tool, separate
  scope.
- `iptables-nft` compatibility tables. They show up in `inet filter` like
  anything else; we don't translate.
